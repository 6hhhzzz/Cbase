# IWMS 第三方服务选型对比

## 文档信息

- **作者**: 李明辉（技术负责人）
- **日期**: 2026 年 4 月 10 日
- **目的**: 为技术架构选型提供量化决策依据

## 一、OCR 服务选型

IWMS 入库环节需要对供应商标签进行 OCR 识别，提取商品信息。评估三家供应商：

| 评估维度 | 百度 OCR API | 阿里云 OCR | 自训练 PaddleOCR |
|---------|:-----------:|:--------:|:--------------:|
| 通用文字识别率 | 94.7% | 92.3% | 89.1%(基础模型) |
| 中文倾斜矫正 | 优秀 | 良好 | 需调参 |
| 破损标签识别 | 良好 | 一般 | 需大量训练数据 |
| API 延迟(P99) | 800ms | 1.2s | 300ms(本地部署) |
| 价格(/万次) | ¥12 | ¥15 | 免费(GPU成本另计) |
| 技术支持和文档 | 完善 | 完善 | 社区支持 |
| 集成难度 | 1天 | 1天 | 1周+ |

**结论**: 选择 **百度 OCR API**。虽然自训练方案价格最低，但耗费时间较长且识别率不确定。百度 OCR 在 POC 测试中识别率达 94.7%，满足 95% 的合同要求（差距 0.3% 通过灰度处理弥补）。

## 二、消息队列选型

IWMS 采用异步消息处理订单流转和 ERP 同步。评估三款 MQ：

| 评估维度 | RabbitMQ 3.x | Apache Kafka | Redis Streams |
|---------|:-----------:|:-----------:|:-----------:|
| 消息可靠性 | 极高(确认机制) | 高(副本) | 中(内存限制) |
| 延迟 | < 1ms | < 10ms | < 1ms |
| 吞吐量 | 20K msg/s | 100K msg/s | 50K msg/s |
| 管理界面 | 完善 | 需额外安装 | 简单 |
| 运维复杂度 | 低 | 高 | 低 |
| 消息重试 | 内置(死信) | 需自实现 | 需自实现 |
| 团队经验 | 现有项目已用 | 无 | 无 |

**结论**: 选择 **RabbitMQ 3.x**。团队已有使用经验，消息可靠性高，管理界面友好。IWMS 对吞吐量要求不高（日均 < 5 万条 MQ 消息），RabbitMQ 完全满足。

## 三、对象存储选型

用于存放入库单扫描件、盘点照片、报表导出文件。评估三款：

| 评估维度 | MinIO | 阿里云 OSS | 自建 NFS |
|---------|:-----:|:--------:|:------:|
| S3 API 兼容 | 完整 | 完整 | 不支持 |
| 私有化部署 | ✅ | ❌ | ✅ |
| 企业版价格 | ¥18,000/年 | ¥0.12/GB/月 | 硬件成本 |
| 高可用 | 集群模式 | 多AZ | 需额外配置 |
| 扩容 | 在线扩容 | 自动 | 停机 |
| 图形化管理 | ✅ | ✅(Web控制台) | ❌ |

**结论**: 选择 **MinIO**。客户要求数据不出私有网络，MinIO 支持私有化部署且 S3 API 兼容。选择企业版许可（¥18,000/年），包含技术支持。

## 四、最终选型汇总

| 组件 | 选型 | 许可/价格 | 选型依据 |
|------|------|---------|---------|
| OCR 服务 | 百度 OCR API | ¥12/万次 | 识别率最高 |
| 消息队列 | RabbitMQ 3.x | 开源免费 | 团队经验 + 可靠性 |
| 对象存储 | MinIO | ¥18,000/年 | 私有化部署要求 |
| 数据库 | PostgreSQL 16 | 开源免费 | 架构设计文档已确定 |
| 缓存 | Redis 7.x | 开源免费 | 架构设计文档已确定 |
