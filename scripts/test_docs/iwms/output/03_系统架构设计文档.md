## 一、架构概述

IWMS 采用**微服务 + 前后端分离**架构。总体分为四层：

```
┌─────────────────────────────────────────────┐
│              前端层 (Vue 3)                   │
│   PC Web (Element Plus) + PDA (H5)           │
├─────────────────────────────────────────────┤
│              API 网关 (Nginx)                 │
│         路由 / 限流 / 负载均衡                 │
├─────────────────────────────────────────────┤
│              微服务层 (Spring Boot 3.3)       │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐       │
│  │入库服务│ │出库服务│ │库存服务│ │权限服务│       │
│  └──────┘ └──────┘ └──────┘ └──────┘       │
│  ┌──────┐ ┌──────┐ ┌──────┐                │
│  │报表服务│ │系统服务│ │消息服务│                │
│  └──────┘ └──────┘ └──────┘                │
├─────────────────────────────────────────────┤
│              中间件层                         │
│  PostgreSQL 16 + Redis 7 + RabbitMQ 3        │
│  MinIO (对象存储)                             │
├─────────────────────────────────────────────┤
│              基础设施层                        │
│  Docker + Kubernetes (客户方提供)             │
└─────────────────────────────────────────────┘
```

## 三、微服务划分

### 3.1 入库服务 (wms-inbound)

- **职责**：采购入库、退货入库、调拨入库、质检管理、上架管理
- **数据库**：`wms_inbound` schema
- **对外 API**：12 个 REST 端点
- **依赖**：库存服务（更新库存）、权限服务（鉴权）

### 3.2 出库服务 (wms-outbound)

- **职责**：销售出库、波次拣货、复核打包、快递面单、出库审批
- **数据库**：`wms_outbound` schema
- **对外 API**：15 个 REST 端点
- **依赖**：库存服务（锁定/扣减库存）、权限服务（审批流）

### 3.3 库存服务 (wms-inventory)

- **职责**：实时库存查询、库存预警、盘点管理、批次追溯
- **数据库**：`wms_inventory` schema
- **对外 API**：8 个 REST 端点
- **依赖**：无（核心服务，被其他服务调用）

### 3.4 权限服务 (wms-auth)

- **职责**：用户认证、RBAC 权限控制、审计日志
- **数据库**：`wms_auth` schema
- **对外 API**：10 个 REST 端点
- **依赖**：Redis（Token 缓存）

### 3.5 报表服务 (wms-report)

- **职责**：报表生成、数据统计、Excel/PDF 导出
- **数据库**：只读访问各 schema 的视图
- **对外 API**：6 个 REST 端点

### 3.6 消息服务 (wms-message)

- **职责**：异步消息处理、ERP 数据同步、库存同步
- **消息队列**：RabbitMQ
- **主要消费者**：ERP 同步、库存变更通知、审批通知

## 五、API 设计原则

- 所有 API 遵循 RESTful 风格，URL 格式：`/api/v1/{resource}`
- 请求和响应统一使用 JSON 格式
- 认证通过 `Authorization: Bearer <token>` 头传递
- 分页参数统一为 `page` 和 `size`，返回 `{items, total, page, size}`
- 错误响应统一为 `{error, message, timestamp}`

完整 API 规范见《REST API 接口规范》文档。

## 六、部署架构

- 每个微服务独立 Docker 镜像，通过 Docker Compose 编排
- 客户方提供 Kubernetes 集群（3 个 Worker 节点），生产环境使用 K8s 部署
- Nginx 作为入口网关，处理 SSL 终结和路由转发
- PostgreSQL 使用主从复制，Redis 使用哨兵模式保证高可用

部署详情见《部署运维手册》文档。

## 七、安全设计

- JWT 双 Token 机制：Refresh Token（7 天）+ Access Token（30 分钟）
- API 调用需携带 Access Token，过期后用 Refresh Token 换发
- 所有敏感操作写入审计日志，日志保留期 12 个月
- 生产环境数据库密码通过 K8s Secret 管理，不写入配置文件
