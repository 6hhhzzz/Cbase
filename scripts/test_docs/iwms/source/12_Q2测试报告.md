## 三、严重缺陷详情

### P0-1: 波次拣货在订单量 > 500 时系统卡死

- **模块**: 出库管理
- **发现日期**: 2026-06-20
- **描述**: 创建波次含 512 个订单时，拣货路径算法计算超时（>45秒），前端出现白屏
- **影响**: 大波次场景下出库流程中断
- **根因**: 路径优化算法（Dijkstra 变体）时间复杂度 O(n²)，n > 500 时不可接受
- **解决方案**: 引入 Redis 缓存货位距离矩阵，计算时间降至 3 秒。同时临时限制波次上限为 300 单。
- **状态**: 已修复（7月24日验证通过）

### P0-2: OCR 识别返回空结果导致入库单无法创建

- **模块**: 入库管理
- **发现日期**: 2026-06-12
- **描述**: 当供应商标签上有污渍或褶皱时，OCR 返回空字符串，系统未处理该异常，导致创建入库单请求超时
- **影响**: 部分供应商无法通过扫码入库
- **根因**: 第三方 OCR 服务对破损标签识别率仅 82%；系统未对空结果做 fallback 处理
- **解决方案**: 切换至百度 OCR API（识别率 94.7%），增加手动输入 fallback 通道
- **状态**: 已修复

### P1-1: 库存扣减在高并发下出现超卖

- **模块**: 出库管理
- **发现日期**: 2026-06-22
- **描述**: 压力测试中，100 QPS 并发出库时，库存扣减出现负库存。经排查，锁定库存和实际扣减之间存在竞态条件。
- **影响**: 严重——可能导致同一件商品被两个订单同时卖出
- **解决方案**: 将库存扣减改为 PostgreSQL `SELECT ... FOR UPDATE` 行级锁，在事务内原子完成
- **状态**: 已修复，回归测试通过
