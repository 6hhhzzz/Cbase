## 二、成功经验

### 2.1 架构设计合理

前期投入 2 个月进行需求分析和系统设计，微服务划分清晰，模块间耦合度低。在入库模块 OCR 服务切换（第三方 → 百度 API）时，修改仅影响入库服务，其他模块未受影响。

### 2.2 敏捷迭代 + 周例会有成效

每两周 Sprint Review + 每周五周例会，问题及时发现和调整。例如 6 月入库模块延期，通过周例会快速决策 OCR 切换方案，将损失控制在 2 周内。

### 2.3 离线模式成为亮点

PDA 离线模式这个需求是客户方在评审中追加的（见《需求规格说明书》3.2 节），虽然增加了 3 周工作量，但最终成为系统最大差异化亮点。竞品中无一支持（见《竞品分析报告》）。

### 2.4 新人导师制有效

应届生钱进通过赵云 mentor 辅导，从 6 月开始代码质量明显提高。PR 平均修改次数从 3 次降至 8 月的 1.2 次。

## 三、踩坑记录

### 3.1 OCR 服务选型过于仓促

最初选择的第三方 OCR 服务仅做了简单 Demo 测试即签约，未充分验证破损标签识别率。导致 6 月紧急切换至百度 OCR API，浪费外包费用 3 万元 + 2 周开发时间。

教训：关键技术选型必须在接近真实场景下做充分 POC。

### 3.2 前端人力预估不足

PDA 适配工作量被严重低估，刘洋在 6 月周会明确提出前端人力紧张（见《6月周会纪要》6/26 记录）。幸好周杰 7 月 1 日入职，及时补上缺口。如果招聘延迟，报表模块可能无法按时交付。

教训：移动端/H5 适配工作量应为 PC Web 的 1.5 倍估算。

### 3.3 库存扣减并发问题

Q2 测试中发现的库存超卖问题（见《Q2测试报告》P1-1），根因是高并发下未加行级锁。这是后端开发的基础问题，说明 Code Review 中对关键业务逻辑的并发安全性审查不够严格。

教训：涉及金钱/库存的关键操作，强制 Code Review 检查并发安全性。

### 3.4 系统管理模块被牺牲

系统管理模块因优先级较低，在 7 月底被缩小范围，部分功能（如系统配置管理）最终未完成，留待二期开发。如果前期更合理地分配资源，可以避免这种"捡了芝麻丢了西瓜"的情况。

## 四、改进建议

### 4.1 技术方面

1. **建立性能基线**：每次发布前自动运行性能回归测试（已在 Q2 测试报告中建议）
2. **增强测试覆盖**：核心业务流程（入库→库存→出库）的端到端测试应在开发阶段就建立，而不是等到 QA 阶段才发现问题
3. **统一异常处理**：各微服务异常响应格式统一，避免前端需要处理多种错误格式

### 4.2 流程方面

1. **POC 评审制度化**：关键技术选型的 POC 必须在评审会上展示真实场景测试结果
2. **资源 buffer**：项目中应有 10-15% 的人力 buffer 应对突发需求，这次 PDA 适配新增需求几乎耗尽全部 buffer
3. **文档同步更新**：在 OCR 切换、PDA 需求变更时，相关文档没有及时更新，给后期运维接手带来困扰

### 4.3 团队方面

1. 钱进的技术成长是本次项目的一个意外收获，建议下个项目继续 mentor 制度
2. 周杰入职一个月即独挡 PDA 端开发，说明社招有经验的人才是应对紧急需求的有效手段
3. 跨部门协作（特别是与客户方 ERP 团队）的沟通成本被低估，建议下次在合同中明确接口人的响应 SLA
